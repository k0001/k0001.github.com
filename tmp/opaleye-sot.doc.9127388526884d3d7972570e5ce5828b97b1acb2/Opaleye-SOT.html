Description

For a better experience, it is recommended that you import this module unqualified as follows:

import Opaleye.SOT

Note that Opaleye.SOT re-exports all of Opaleye.SOT.Run, you might want to refer to that module for documentation.

Both Opaleye.SOT and Opaleye.SOT.Run override some of the names exported by the Opaleye, so it is recommended that you import Opaleye, if needed, qualified as:

import qualified Opaleye as O

This module doesn't export any infix operator.

Defining a Tabla

class ITabla t => Tabla t

Tabla means table in spanish.

An instance of this class can uniquely describe a PostgreSQL table and how to convert back and forth between it and its Haskell representation used when writing Opaleye queries.

The t type is only used as a tag for the purposes of uniquely identifying this Tabla.

Associated Types

type Database t :: *

Some kind of unique identifier used for telling appart the database where this table exists from other databases, so as to avoid accidentally mixing tables from different databases in queries.

type SchemaName t :: Symbol

PostgreSQL schema name where to find the table (defaults to "public", PostgreSQL's default schema name).

type TableName t :: Symbol

Table name.

type Cols t :: [Col Symbol WD RN * *]

Columns in this table. See the documentation for Col.

Working with Tabla

Build the Opaleye Table for a Tabla.

queryTabla :: Tabla t => T t -> Query (PgR t)

Like opaleye's queryTable, but for a Tabla.

type HsR t = Rec t (Cols_HsR t)

Expected output type for runQuery on a (PgR t).

Important: If you are expecting a (PgR t) on the right side of a leftJoin, you will need to use (Maybe (PgR t)).

Mnemonic: Haskell Read.

type HsI t = Rec t (Cols_HsI t)

HsI t is the Haskell representation of Haskell values to be inserted to the database, as taken by "Opaleye.SOT.Run.runInsertTabla".

An HsI t can always be converted to a PgW t using pgWfromHsI, in case you need that for with the more general "Opaleye.SOT.Run.runInsert".

Mnemonic: Haskell Insert.

mkHsI

Arguments

=> T t 
-> ((forall c a. C c -> a -> TCa t c a) -> HList xs) 
-> HsI t 

Helper function to safely build an HsI t.

The type of this function isn't easy to understand, but an example should clarify its usage. Here we asume we have a Person datatype and a TPerson which is an instance of Tisch.

personToHsI :: Person -> HsI TPerson
personToHsI (Person name age) =
   mkHsI (T :: TPerson) $ \set_ -> hBuild
       (set_ (C :: C "name") name)
       (set_ (C :: C "age") age)

You are not required to use this function to build an HsI t if working with the tools from Data.HList is sufficient for you. This is just a convenience.

type PgR t = Rec t (Cols_PgR t)

Output type of queryTabla (T t).

Mnemonic: PostGresql Read.

type PgRN t = Rec t (Cols_PgRN t)

Like (PgRN t) but every field is Koln, as in the output type of the right hand side of a leftJoin with '(table t).

Mnemonic: PostGresql Read Nulls.

Representation of PostgreSQL values to be written to the database. This type can be used as input for "Opaleye.SOT.Run.runInsert" and similar.

Kol

data Kol a

Like opaleye's (Column x), but with x guaranteed to be not Nullable. If you need to have a Nullable column type, use Koln instead.

Build using kol or Kol.

Notice that Kol is very different from Col: Col is used to describe the properties of a column at compile time. Kol is used at runtime for manipulating with values stored in columns.

We do not use (Column x), instead we use (Kol y) where x ~ PgType y. This is where we drift a bit appart from Opaleye. See https://github.com/tomjaguarpaw/haskell-opaleye/issues/97

class PgPrimType p => ToKol a p where

Build a Kol.

You need to provide a ToKol instance for every Haskell type you plan to convert to its PostgreSQL representation as Kol.

A a default implementation of kol is available for Wrapped instances:

default kol :: (PgTyped b, PgType b ~ p, Wrapped a, ToKol (Unwrapped a) p) => a -> Kol b
kol = kol . view _Wrapped'

Implementation notice: This class overlaps in purpose with Opaleye's Constant. Technicaly, we don't really need to repeat those instances here: we could just rely on Opaleye's Constant. However, as of today, Opaleye's Constant provides some undesired support which we deliberately want to avoid here. Namely, we don't want to support converting Int to PGInt4. If this is fixed upstream, we might go back to relying on Constant if suitable. See https://github.com/tomjaguarpaw/haskell-opaleye/pull/110

Koln

data Koln a

Like opaleye's Column (Nullable x), but with x guaranteed to be not-Nullable.

Think of Koln a as Maybe (Kol a), with nul being analogous to the Nothing constructor and koln being analogous to the Just constructor.

Build safely using nul, koln, fromKol or Koln.

Notice that Koln is very different from Col: Col is used to describe the properties of a column at compile time. Koln is used at runtime for manipulating with values stored in columns.

We do not use Column (Nullable x), instead we use Koln y where x ~ PgType y. This is where we drift a bit appart from Opaleye. See https://github.com/tomjaguarpaw/haskell-opaleye/issues/97

Types

data Col name wd rn pgType hsType

Column description.

This is only used as a promoted datatype expected to have kind Col Symbol WD RN * *.

  • name: Column name.
  • wd: Whether DEFAULT can be written to this column (WD) or not (W).
  • rn: Whether NULL might be read from this column (RN) or not (R).
  • pgType: Type of the column value used in Opaleye queries as index to Kol or Koln. This must be an instance of PgTyped.
  • hsType: Type of the column value used in Haskell outside Opaleye queries. Hint: don't use something like Maybe Bool here if you want to indicate that this is an optional Bool column. Instead, use Int here and RN in the rn field.

Notice that Col is very different from Kol and Koln: Kol and Koln are used at runtime for manipulating values stored in columns, Col is used to describe the properties of a column at compile time.

class PgPrimType (PgType a) => PgTyped a

Only PgTyped instances are allowed as indexes to Kol and Koln.

Associated Types

type PgType a :: *

PgType a indicates the primitive PostgreSQL column type that will ultimately be used as the index to opaleye's Column. This could be a itself, in the case of primitive types such as PGInt4, or it could be something else.

The motivation for this is the same as for “newtype wrappers” in Haskell. Say in you have this newtype in Haskell:

newtype UserId = UserId Int32

Now UserId and Int32 are different types, even though they share the same underlying representation Int32. With PgTyped you can achieve something similar:

instance PgTyped UserId where
  type PgType UserId = PGInt4

With that in place, you won't be able to accidentally mistake Kol PGInt4 for Kol UserId values, yet you will be able to easily reuse most of the machinery available to Kol PGInt4, except:

  • You will need a QueryRunnerColumnDefault instance to fetch Kol UserId values from the database as UserId:
  instance QueryRunnerColumnDefault PGInt4 UserId

You might find queryRunnerColumnFromWrapped useful for simple cases like UserId.

Notice that the instance mentions PGInt4 directly, not our PgTyped UserId. This is fine, not much would be gained by making the difference between them at this point, as you still need to do the FromRow parsing and any inconsistencies will be uncovered there.

  • If you want to reuse the exising Num or Fractional instances for Kol PGInt4, you will need to explicitely ask for it by instantiating PgNum UserId and/or PgFractional UserId. Most likely you won't need this for cases such as UserId@, since you shouldn't be doing arithmetic with user identifiers anyway.
